# 採購類別欄位：失敗原因與欄位建議

## 剛剛失敗的可能原因

1. **程式先假設「關聯表」**
   - 程式一開始是查獨立的 `main_category` 表，或使用 `vendors.main_category(id, name)` 這種「關聯展開」語法。
   - 若你的 `vendors` 表只有 **`main_category` 一個欄位（varchar）**，沒有 `main_category_id` 也沒有對應的類別表，Supabase 會報錯或回傳 null，類別下拉就會是空的。

2. **Supabase 關聯語法的前提**
   - `main_category(id, name)` 這種寫法只適用在：`vendors` 有 **外鍵** `main_category_id` 指向另一個表（例如 `main_category` 表）。
   - 若 `main_category` 只是 `vendors` 表上的一個 **文字欄位（varchar）**，就沒有「關聯」可展開，查詢會失敗或拿不到資料。

3. **欄位名稱或權限**
   - 若實際欄位名稱不是 `main_category`（例如是 `category` 或 `main_category_name`），查詢也會失敗。
   - RLS 權限若不允許讀取該欄位，同樣會拿不到資料。

---

## 欄位可以設成 varchar 嗎？

**可以。** 類別用 **varchar 存名稱** 是合理做法，程式已支援：

- 現在會**優先**從 `vendors.main_category`（當成一般欄位）讀取，取不重複值當作類別清單。
- 若你沒有 `main_category_id`、也沒有類別表，只要 `vendors` 有 `main_category`（varchar），類別下拉就應能正常顯示。

請確認：

- 表名：`vendors`
- 欄位名：`main_category`（型別 varchar/text）

若你的欄位名不同，告訴我實際名稱，我可以幫你對應到程式裡。

## 欄位選擇建議：varchar  vs  獨立類別表 + 外鍵

| 做法 | 優點 | 缺點 |
|------|------|------|
| **varchar（目前做法）** | 結構簡單、不用多一張表、查詢直覺、程式已支援 | 類別名稱可能不統一（例如「塗料」「塗料類」）、日後要改名稱要更新多筆 |
| **獨立類別表 + 外鍵** | 類別名稱一致、可集中改名稱、可加排序/說明等欄位、報表與篩選較穩 | 需要多一張表（如 `main_category`）和 `vendors.main_category_id`，結構稍複雜 |

**建議：**

- **先維持 varchar**：若目前資料量不大、類別名稱會手動控制，用 `vendors.main_category`（varchar）即可，程式已依此優化。
- **之後再正規化**：若發現類別名稱重複、不一致或常要改，再在 Supabase 新增類別表（如 `main_category(id, name)`），並在 `vendors` 加 `main_category_id` 外鍵，程式也已有關聯查詢的備援邏輯可接。

## 若類別還是沒出現，請檢查

1. **Supabase → Table Editor → vendors**
   - 是否有欄位 **`main_category`**（varchar/text）？
   - 裡面是否有幾筆有填類別名稱（非空）？

2. **RLS**
   - 該專案使用的 Supabase 角色，是否有 **select** `vendors` 的權限（至少包含 `main_category`）？

3. **瀏覽器開發者工具 (F12) → Network**
   - 重新打開「新增進貨紀錄」、選類別時，看對 Supabase 的請求是否成功、回應裡是否有 `main_category` 的資料。
